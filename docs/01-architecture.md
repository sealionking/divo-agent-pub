# 体系架构

> 本文描述 DiVo Agent Platform 的功能架构，**不涉及任何具体技术栈或上游代码引用**。

---

## 设计哲学

DiVo Agent Platform 围绕三个不可让步的原则构建：

### 1. 环境即治理单元
治理的最小粒度不是 Agent，而是 Agent **所在的工作空间**。策略绑定在环境上，Agent 跑在环境里。换 Agent 不改策略；改策略不用管里面跑的是哪个 Agent。

### 2. 零信任出站
工作空间默认对互联网封闭。所有出站调用（LLM API、MCP 代理、插件安装、文件上传）都必须走平台声明的**出站网关**。白名单之外一律拒绝、日志留痕。

### 3. 配置即资产
工作空间定义、策略、Skill 清单、审计配置 —— 所有都纳入版本控制，纳入审计流水。"谁在什么时候让某个请求通过了"，永远可回查。

### 4. 策略即边界
**"限制策略就是限制 Agent 的行为边界"** —— IT 只定义"允许做什么"，剩下的 Agent 自己发挥。具体在架构上怎么实现策略容器化、版本化、一键生效，这是 DiVo 内部的核心竞争力，不在此处展开。

## 数据模型

```
TemplateVersion ──1─*── WorkspaceBuild ──1─*── WorkspaceResource
                     │
                     ├── WorkspaceAgent (runs in container)
                     │        │
                     │        └── 出站请求 ──> LLM Relay / MCP Hub
                     │
                     └── AuditLog (every call, every decision)

IdentityProvider (OIDC/SAML/CAS) ──1─*── User ──*─*── Workspace
```

- **TemplateVersion** 描述一类工作空间"该长什么样"
- **Workspace** 是某个用户按模板领到的具体工作空间
- **AuditLog** 是不可篡改的追加日志，留存留痕
- **IdentityProvider** 不限定具体产品，任何兼容 OIDC / SAML / CAS 的 IdP 都能接入

## 网络边界

| 方向 | 默认策略 | 可配置 |
|---|---|---|
| 工作空间 -> 互联网 | ❌ 拒绝 | 通过 LLM Relay 代理出站 |
| 工作空间 -> 私网 | ❌ 拒绝 | 通过白名单 IP/域名段放行 |
| 控制平面 -> 工作空间 | ✅ 单向下发 | 反向需经 Authentication |
| 工作空间 ↔ 工作空间 | ❌ 隔离 | 仅 MCP Hub 内服务互通 |
| Web 终端 <- 用户 | ✅ HTTPS WSS | 强制 IdP 认证 |

---

## 关键延迟路径

1. **用户发起请求** → IdP 校验 JWT → 控制平面鉴权 → 路由到工作空间 Agent
2. **Agent 调用 LLM** → 出站网关策略匹配 → LLM Relay 转发 → 模型返回 → 落审计
3. **Agent 调用工具 / MCP** → 出站网关策略匹配 → 内网 MCP Hub → 审计

任何一步策略不匹配，请求拒绝 + 告警 + 人工复核队列。
